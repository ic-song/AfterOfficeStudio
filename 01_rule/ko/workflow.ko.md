# AI Workflow 한국어 검토본

이 문서는 `01_rule/ai/workflow.md`의 사람용 한국어 검토본입니다.

## 1. 목적

아이디어가 프로젝트 후보가 되고, 필요한 경우 실제 프로젝트 문서로 정리되는 흐름을 정의합니다.

## 2. 기본 흐름

```text
1. 아이디어 수집
2. Swain이 방향성과 프로젝트화 가능성 검토
3. Shen이 작업 흐름 정리
4. Lux가 기획 초안 작성
5. 필요 시 BM 가능성 검토
6. Viktor가 기술 가능성 검토
7. Senna가 상태와 기록 정리
8. 필요 시 50_프로젝트/ 아래에 프로젝트 생성
9. 최종 결정은 DECISIONS.md에 기록
```

## 3. 단계별 기준

- 아이디어 수집: 배경, 문제, 예상 사용자, 현재 상태를 정리합니다.
- Swain 방향성 검토: 프로젝트로 관리할 가치가 있는지 판단합니다.
- Shen 작업 흐름 정리: 다음 액션, 담당 agent, 필요한 문서, 진행 순서를 정리합니다.
- Lux 기획 초안 작성: 문제 정의, 대상 사용자, MVP 범위, 요구사항을 작성합니다.
- BM 가능성 검토: 확정이 아니라 가능성 중심으로 정리합니다.
- Viktor 기술 가능성 검토: 구현 가능성, 구조, 리스크, 테스트 기준을 검토합니다.
- Senna 상태와 기록 정리: 상태, 결정사항, 문서 최신화 필요 여부를 정리합니다.

## 4. 상태 기준

```text
IDEA    : 아이디어 단계
TODO    : 진행 후보
READY   : 기획 또는 작업 준비 완료
DOING   : 진행 중
HOLD    : 보류
LIVE    : 운영 반영 완료
DONE    : 종료
ARCHIVE : 보관
```

상태 변경 시 변경 이유를 함께 기록합니다.
## 추가 규칙: Senna 회의록 정리

회의나 논의를 운영 기록으로 남겨야 할 때 Senna는 `01_rule/ko/meeting-note-rules.ko.md`를 따릅니다.

전체 운영 또는 프로젝트 미지정 회의록은 `90_기록실/회의록/`에 저장하고, 특정 프로젝트 회의록은 `50_프로젝트/{project}/meetings/`에 저장합니다.

회의록은 대화 흐름, 주요 의견, 주요 쟁점, 결정사항, 보류사항, 다음 작업을 보존해야 합니다.

녹취록이 필요한 경우 별도 파일로 만들지 않고 같은 회의록 파일 안에 `녹취록` 섹션으로 포함합니다.

Agent 발언자는 파일명이나 경로가 아니라 별칭으로 표기하며, 별칭과 역할 기준은 `01_rule/agents/core/role-map.md`를 따릅니다.
