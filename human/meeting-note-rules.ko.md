# 회의록 작성 규칙

## 목적

After Office Studio의 회의록 작성 기준을 정리한다.

회의록은 사람이 보는 운영 기록 문서이며, 회의나 논의의 대화 흐름, 주요 의견, 결정사항, 보류사항, 다음 작업을 남기는 문서로 정의한다.

## 기본 원칙

회의록은 단순히 결론만 남기는 문서가 아니다.

회의록은 논의 과정에서 어떤 이야기가 오갔고, 어떤 의견을 거쳐 어떤 결정이 났으며, 다음에 무엇을 해야 하는지 추적하기 위한 운영 기록이다.

회의록은 기본적으로 하나의 회의록 파일로 저장하며, 필요한 경우 회의록 안에 `녹취록` 섹션을 포함한다.

## 회의록 정의

회의록은 회의나 논의 내용을 주제별로 정리한 운영 기록 문서이다.

회의록은 다음 내용을 중심으로 작성한다.

- 논의 주제
- 대화 요약
- 주요 의견
- 주요 쟁점
- 결정사항
- 보류사항
- 다음 작업

## 녹취록 정의

녹취록은 회의나 논의 중 오간 대화 흐름과 주요 발언을 원문에 가깝게 기록한 섹션이다.

After Office Studio에서는 녹취록을 별도 문서로 저장하지 않고, 필요한 경우 회의록 파일 안에 `녹취록` 섹션으로 포함한다.

녹취록은 모든 문장을 반드시 완전한 원문 그대로 남기는 것이 아니라, 나중에 대화 맥락을 이해할 수 있도록 주요 발언과 흐름을 보존하는 데 목적이 있다.

## 회의록과 녹취록의 관계

회의록과 녹취록은 별도 파일로 분리하지 않는다.

하나의 회의록 파일 안에서 다음 두 영역을 함께 관리한다.

```text
회의록 = 논의 내용을 정리한 운영 기록
녹취록 = 대화 흐름과 주요 발언을 원문에 가깝게 남긴 기록
```

녹취록은 필요한 경우에만 작성한다.

## 파일명 규칙

회의록은 주제별로 정리하며, 파일명에는 문서명 뒤에 작성 일자를 명시한다.

파일명 형식은 다음을 따른다.

```text
{주제명}-회의록-{YYYY-MM-DD}.md
```

예시:

```text
에이전트-구조-논의-회의록-2026-05-20.md
회의록-정의-회의록-2026-05-20.md
아이디어-선별-사이클-회의록-2026-05-20.md
```

동일한 주제로 같은 날 여러 회의록이 생기면 뒤에 번호를 붙인다.

```text
에이전트-구조-논의-회의록-2026-05-20-01.md
에이전트-구조-논의-회의록-2026-05-20-02.md
```

이 파일명 규칙은 회의록 운영 문서에 대한 명시적 예외이다. 일반 한국어 문서의 `.ko.md` suffix 규칙과 충돌할 때 회의록 파일에는 이 회의록 파일명 규칙을 우선 적용한다.

## 저장 위치

회의록은 주제의 소속에 따라 저장 위치를 나눈다.

After Office Studio 전체 운영, Agent 구조, 운영 규칙, 아이디어 선별, 워크스페이스 관리처럼 특정 프로젝트에 속하지 않는 회의록은 `human/meetings/`에 저장한다.

특정 프로젝트의 기획, 결정, 구현, 회고, 상태 논의처럼 프로젝트에 직접 연결된 회의록은 `projects/{project}/meetings/`에 저장한다.

기본 규칙:

```text
전체 운영 또는 프로젝트 미지정 회의록
-> human/meetings/

특정 프로젝트 회의록
-> projects/{project}/meetings/
```

예시:

```text
human/meetings/에이전트-구조-논의-회의록-2026-05-20.md
human/meetings/회의록-정의-회의록-2026-05-20.md
human/meetings/운영-규칙-정리-회의록-2026-05-20.md
```

```text
projects/my-app/meetings/MVP-범위-논의-회의록-2026-05-20.md
projects/my-app/meetings/기술-구조-검토-회의록-2026-05-20.md
projects/my-app/meetings/릴리즈-준비-회의록-2026-05-20.md
```

녹취록은 별도 저장 위치를 갖지 않는다. 필요한 경우 해당 회의록 파일 내부의 `녹취록` 섹션에 포함한다.

회의록에서 확정된 중요한 결정은 관련 프로젝트의 `DECISIONS.md` 또는 적절한 결정 로그에도 요약 반영한다.

## 작성 언어 기준

회의록은 사람이 보는 운영 기록 문서이므로 한글 위주로 작성한다.

영문 용어, 파일명, 경로, 명령어, 코드명, Agent 이름 등은 필요한 경우에만 원문을 유지한다.

## Agent 발언자 표기 기준

회의록에서 Agent의 발언이나 의견을 기록할 때는 Agent 파일명 전체나 경로가 아니라 별칭을 사용한다.

Agent 별칭과 역할 기준은 `agents/core/role-map.md`를 따른다.

회의록 문서 안에는 Agent 별칭 목록을 직접 하드코딩하지 않는다.

Agent 별칭이나 역할이 변경되는 경우 회의록 정의 문서가 아니라 `agents/core/role-map.md`를 먼저 갱신한다.

## Agent 참조 기준

다른 문서에서 Agent를 참조하거나 역할을 정의할 때는 `agents/core/role-map.md`를 우선 기준으로 삼는다.

Agent 별칭, 역할, 담당 영역, 관련 파일 경로는 각 문서에 중복 작성하지 않고 `agents/core/role-map.md`를 참조한다.

## 회의록 기본 구조

회의록은 아래 구조를 기본으로 한다.

```markdown
# {주제명} 회의록

---

## 작성 정보

- 작성 일자:
- 관련 주제:
- 관련 Agent:

---

## 회의 목적

---

## 대화 요약

---

## 주요 의견

---

## 주요 쟁점

---

## 결정사항

---

## 보류사항

---

## 다음 작업

---

## 녹취록

필요한 경우에만 작성한다.

Agent 발언은 별칭으로 작성한다.

예시:

Swain:
현재 Agent 구조는 조직도형으로 재배치하는 방향이 적절하다.

Shen:
이번 작업은 전체 프로젝트 구조 변경이 아니라 agents/ 하위 구조 정리로 제한한다.
```

## 작성 기준

- 회의록은 사람이 보는 운영 기록 문서로 작성한다.
- 회의록 본문은 한글 위주로 작성한다.
- 회의록은 주제별로 정리한다.
- 서로 다른 주제를 하나의 회의록에 무리하게 섞지 않는다.
- 회의록 파일명에는 문서명 뒤에 작성 일자를 명시한다.
- 전체 운영 또는 프로젝트 미지정 회의록은 `human/meetings/`에 저장한다.
- 특정 프로젝트 회의록은 `projects/{project}/meetings/`에 저장한다.
- 녹취록은 별도 파일로 분리하지 않고 회의록 내부 섹션으로 포함한다.
- 녹취록 섹션은 필요한 경우에만 작성한다.
- Agent 발언은 파일명이나 경로가 아니라 별칭으로 작성한다.
- Agent 별칭 기준은 `agents/core/role-map.md`를 따른다.
- 회의록 문서 안에 Agent 별칭 목록을 하드코딩하지 않는다.
- 파일명, 경로, 명령어, 코드명은 필요한 경우 원문을 유지한다.
- 한글 문서는 UTF-8 기준으로 작성한다.

## 회의록 발언 표기 예시

좋은 예:

```text
Swain:
현재 Agent 구조는 조직도형으로 재배치하는 방향이 적절하다.

Shen:
이번 작업은 전체 프로젝트 구조 변경이 아니라 agents/ 하위 구조 정리로 제한한다.
```

피하는 예:

```text
swain-core-agent.md:
현재 Agent 구조는 조직도형으로 재배치하는 방향이 적절하다.

teams/workspace/sona-library-agent.md:
기존 문서와 중복되는 회의록 정의 파일이 있는지 먼저 확인해야 한다.
```

## 주의사항

- 기존 회의록 정의 문서가 있다면 새 문서를 중복 생성하지 말고 기존 문서에 추가한다.
- 기존에 녹취록 별도 저장 규칙이 있다면, “회의록 내부 녹취록 섹션으로 포함”하는 방식으로 수정한다.
- Agent 별칭 목록을 회의록 정의 문서에 하드코딩하지 않는다.
- Agent 별칭과 역할 기준은 `agents/core/role-map.md`에 의존한다.
